# РЛК «Исток» – распределённый комплекс визуализации радиолокационных портретов

Дипломный проект: **«Разработка программных решений для АО «НПП «Исток» им. Шокина» в рамках автоматизации управления жизненным циклом специализированных аппаратных комплексов. Разработка распределенного программного комплекса визуализации и анализа радиолокационных портретов в системе радиоэлектронного мониторинга»**.

## Архитектура (микросервисы)

Проект реализован как **три независимых микросервиса**, взаимодействующих по протоколам HTTP/WebSocket:

| Сервис       | Порт | Назначение                                                                       |
|--------------|------|----------------------------------------------------------------------------------|
| **Storage**  | 8081 | Хранение данных (MySQL через GORM). REST API для сохранения и получения истории. |
| **Gateway**  | 8080 | WebSocket-сервер, маршрутизация сообщений, broadcast клиентам.                   |
| **Web**      | 8082 | Отдача статических файлов (HTML, CSS, JS).                                       |

Компоненты могут масштабироваться независимо. Gateway не имеет прямого доступа к БД – вся работа с данными делегируется Storage через HTTP. Внутри `storage` реализован отдельный слой репозитория в `internal/repository`.

### Отдельный сервис хранения данных
Сервис `storage` является отдельным микросервисом, ответственным за работу с базой данных. Он выступает в роли репозитория: принимает данные от `gateway`, сохраняет их в MySQL и возвращает историю по запросу. В этом проекте `storage` и есть выделенный сервис для работы с БД.

## Быстрый запуск (3 терминала)

### 1. Подготовка
- Установите Go 1.25+ и MySQL.
- Создайте базу данных `RTL_SDR_DB`.
- Скопируйте `.env.example` в `.env` и заполните свои значения.
- Файл `.env` находится в `.gitignore`, поэтому он не попадёт в репозиторий.

### 2. Запуск микросервисов
```bash
# Терминал 1 – Storage API
go run ./cmd/storage

# Терминал 2 – WebSocket Gateway
go run ./cmd/gateway

# Терминал 3 – Web-сервер статики
go run ./cmd/web
## Открытие интерфейса
Перейдите в браузере по адресу: http://127.0.0.1:8082

WebSocket-клиент автоматически подключается к ws://127.0.0.1:8080/ws.

Протокол обмена (WebSocket, JSON)
Все сообщения обёрнуты в Envelope:

json
{
  "type": "detection_data",
  "payload": { ... }
}
Типы сообщений
Тип	Направление	Описание
detection_data	Внешний алгоритм → Gateway	Передача радиолокационного портрета (ваш JSON). Gateway сохраняет через Storage и рассылает всем клиентам.
new_detection_data	Gateway → клиенты	Broadcast-сообщение о новом обнаружении.
command	Клиент → Gateway	Выполнение команды (ping, get_history и др.).
get_history	Клиент → Gateway	Запрос истории обнаружений (ответ приходит как get_history_response).
error	Gateway → клиент	Сообщение об ошибке валидации/обработки.
Пример внешнего сообщения (detection_data)
Полный пример JSON – см. в файле web/js/app.js (демо-данные) или в документации.

Расчёт FPS (кадры в секунду) и Ping
FPS (радар)
Рассчитывается в модуле web/js/radar.js:

Каждый вызов draw() инкрементирует счётчик кадров frameCount.

Раз в секунду (по таймеру performance.now()) вычисляется:

text
fps = (frameCount * 1000) / elapsedMilliseconds
Значение округляется и выводится в интерфейсе.

Перерисовка радара происходит автоматически при получении новых данных, а также принудительно каждые ~33 мс (setInterval(…, 33)), что обеспечивает плавное отображение.

Ping (задержка WebSocket)
Реализован в web/js/websocket.js:

Клиент каждые 2 секунды отправляет команду {"type":"command","payload":{"command":"ping"}}.

В момент отправки фиксируется pingStartTime = Date.now() и устанавливается флаг pendingPing = true.

Gateway отвечает {"type":"command_response","payload":{"result":"pong"}}.

При получении ответа вычисляется latency = Date.now() - pingStartTime.

Если ответ не пришёл в течение 1 секунды, ping считается потерянным и отображается --.

Результат выводится в блоке PING (мс) интерфейса.

Хранение данных (Storage API)
Storage предоставляет два REST-эндпоинта:

POST /api/detections – сохранение сессии и детекций (принимает тот же JSON, что и detection_data).

GET /api/history?limit=N – получение последних N (не более 100) сессий с детекциями.

Storage логирует session_id, количество детекций, время обработки и образец detection_id для быстрой трассировки.

Gateway не использует GORM и не подключается к MySQL – только HTTP-запросы к Storage.

Веб-интерфейс (статический клиент)
Структура:

web/index.html – основной HTML, содержит сетки и блоки.

web/css/style.css – тактический стиль (тёмно-зелёный, моноширинный, без излишеств).

web/js/radar.js – класс Radar: отрисовка полярной сетки, целей, обработка масштаба/панорамирования, расчёт FPS.

web/js/websocket.js – класс RadarWebSocket: управление соединением, отправка команд, измерение ping.

web/js/app.js – главный модуль: связывает радар и WebSocket, обновляет статистику и журнал.

Функциональность интерфейса
Радар: полярная система координат (азимут 0…360°, дальность до 20 км). Масштабирование колёсиком мыши, панорамирование зажатой кнопкой.

Статистика: количество целей, средняя достоверность, доля надёжных целей, время последнего обновления.

Распределение по типам: гистограмма типов объектов (drone, airplane, bird …).

Журнал обнаружений: последние 15 записей с прокруткой, цветовая индикация риска.

FPS и Ping: отображение производительности и качества канала связи.

Разработка и расширение
Добавление нового типа сообщения
В internal/ws/protocol/envelope.go определите константу типа.

Реализуйте обработчик в internal/handlers/ (должен возвращать (interface{}, error)).

В cmd/gateway/main.go зарегистрируйте обработчик: srv.Handle("my_type", handlers.MyHandler(...)).

На клиенте в websocket.js или app.js добавьте обработку входящего сообщения.

Масштабирование микросервисов
Storage можно заменить на любую СУБД (через GORM), поднять несколько реплик с балансировщиком.

Gateway горизонтально масштабируется – клиенты подключаются к любому экземпляру. Для broadcast между экземплярами потребуется message broker (например, Redis Pub/Sub).

Web-сервер статики можно вынести за CDN.

Устранение неполадок
Проблема	Вероятная причина	Решение
Интерфейс не загружается	Не запущен web-сервер (:8082)	Выполните go run ./cmd/web
WebSocket не подключается	Не запущен gateway (:8080)	Выполните go run ./cmd/gateway
Нет истории	Не запущен storage (:8081) или нет данных в БД	Запустите go run ./cmd/storage, отправьте тестовое detection_data
Ping отображается --	Ответ не получен за 1 сек или команда не поддерживается	Проверьте логи gateway, убедитесь, что command.go обрабатывает ping
