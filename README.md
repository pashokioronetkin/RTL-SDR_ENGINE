# РЛК "Исток"

**Распределённый комплекс визуализации и анализа радиолокационных портретов**

Назначение: приём, хранение и отображение радиолокационных портретов, полученных от алгоритмов обнаружения, с передачей информации в реальном времени через WebSocket.

Проект разработан для АО «НПП «Исток» им. Шокина» в рамках дипломного проекта.

---

## Архитектура

Комплекс состоит из трёх независимых микросервисов, взаимодействующих по HTTP и WebSocket:

| Сервис    | Порт (по умолчанию) | Назначение |
|-----------|---------------------|------------|
| Storage   | 8081                | REST API для сохранения и получения данных (MySQL + GORM) |
| Gateway   | 8080                | WebSocket-сервер: маршрутизация сообщений, broadcast клиентам |
| Web       | 8082                | Раздача статического веб-интерфейса |

Сервисы масштабируются независимо. Gateway не имеет прямого доступа к базе данных — все операции с данными делегируются Storage через HTTP.

---

## Требования

- Go 1.21+
- MySQL 8.0 (или совместимая версия)
- Современный браузер для веб-интерфейса

## Быстрый запуск

1. Установите Go и MySQL.
2. Создайте базу данных `RTL_SDR_DB`.
3. Настройте переменные окружения в файле `.env` (см. `.env.example`). Файл `.env` помещён в `.gitignore`.
4. Запустите три компонента в отдельных терминалах:

# Терминал 1: Storage API
go run ./cmd/storage

# Терминал 2: WebSocket Gateway
go run ./cmd/gateway

# Терминал 3: Web-сервер статики
go run ./cmd/web
Откройте в браузере http://127.0.0.1:8082.
WebSocket-клиент автоматически подключится к ws://127.0.0.1:8080/ws.

Протокол обмена (WebSocket)
Все сообщения обёрнуты в универсальный конверт (Envelope):

```bash
{
  "type": "detection_data",
  "payload": { ... }
}```

Типы сообщений
Тип	Направление	Описание
detection_data	Внешний источник -> Gateway	Передача радиолокационного портрета. Gateway сохраняет данные через Storage и рассылает всем клиентам.
new_detection_data	Gateway -> Клиенты	Широковещательное сообщение с новыми данными обнаружения.
command	Клиент -> Gateway	Выполнение команды (ping, get_history и др.). Ответ приходит как command_response.
get_history	Клиент -> Gateway	Запрос истории обнаружений (ответ — get_history_response).
error	Gateway -> Клиент	Сообщение об ошибке.
Пример сообщения detection_data можно найти в web/js/app.js.

# Storage API
Эндпоинты
POST /api/detections — сохранение одной сессии и списка детекций (тело запроса аналогично payload из detection_data).

GET /api/history?limit=N — получение последних N сессий (максимум 100) с вложенными детекциями.

Сервис логирует session_id, количество детекций, длительность операций и идентификаторы первых объектов для трассировки.

# Веб-интерфейс
Структура статических файлов:

web/
├── index.html
├── css/
│   └── style.css
└── js/
    ├── app.js         - инициализация, обновление UI
    ├── radar.js       - отрисовка полярного радара
    └── websocket.js   - WebSocket-клиент с измерением пинга

Функциональные блоки
Полярный радар: азимут 0..360°, дальность до 20 км. Масштабирование колёсиком мыши, панорамирование зажатием. Клик по цели — подробная информация.

Тактическая сводка: общее количество целей, средняя достоверность, процент надёжных объектов, время последнего обновления.

Распределение по типам (drone, airplane, bird и т.д.).

Журнал обнаружений: последние 15 записей, цветовая индикация риска.

FPS — частота перерисовки canvas (вычисляется в radar.js).

Ping (мс) — задержка WebSocket-соединения (измеряется отправкой команды ping каждые 2 секунды).

Расширение системы
Добавление нового типа сообщения
В internal/ws/protocol/envelope.go определите константу типа.

Создайте обработчик в internal/handlers/ (функция должна возвращать (interface{}, error)).

Зарегистрируйте обработчик в cmd/gateway/main.go:
srv.Handle("new_type", handlers.NewHandler(...)).

На клиенте (например, в app.js) добавьте реакцию на сообщения нового типа.

Масштабирование
Storage: горизонтальное масштабирование через репликацию MySQL; возможна замена драйвера GORM на другую СУБД.

Gateway: несколько экземпляров за балансировщиком; для синхронизации broadcast между экземплярами нужен брокер сообщений (Redis Pub/Sub).

Web: без изменений; может быть размещён на CDN.

Устранение неполадок
Проблема	Вероятная причина	Решение
Интерфейс не открывается	Не запущен Web (порт 8082)	go run ./cmd/web
WebSocket не подключается	Не запущен Gateway (порт 8080)	go run ./cmd/gateway
История пуста	Не запущен Storage (порт 8081) или нет данных	Запустите storage, отправьте тестовое detection_data
Ping показывает --	Команда ping не обработана или тайм-аут	Проверьте логи Gateway, убедитесь, что обработчик command поддерживает ping
